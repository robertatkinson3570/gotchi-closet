Logged per the prompt's "ask zero clarifying questions" rule. Date: 2026-07-09.

## Environment and tooling

1. **SiteCrawlIQ full crawl was impractical here, so I ran its audit engines standalone.** The SiteCrawlIQ working tree at `C:\Cursor\SiteCrawlIQ` had been cleared (README said restore via `git checkout .`, which I did). The full platform needs PostgreSQL, Stripe, Resend, and session secrets to run crawls through the normal pipeline (`server/crawler.ts` is coupled to `storage.ts`/Drizzle). Instead I invoked its pure audit modules directly: `runGeoAudit()` from `server/geo-audit.ts` and `validateStructuredData()` from `server/structured-data.ts`, fed with the live raw HTML of 11 gotchicloset.com URLs plus robots.txt. Output: `seo-output/sitecrawliq-audit.json`. The runner source is preserved as `seo-output/sitecrawliq-runner.ts`. This matches SiteCrawlIQ's raw-HTML (non-JS-rendering) crawler view, which is exactly the crawler-visibility question at issue. Core Web Vitals were NOT measured (SiteCrawlIQ's CWV module needs CrUX API access; Lighthouse was out of scope for this pass); performance items in technical-fixes.md are based on code inspection.
2. **Canonical domain is `https://www.gotchicloset.com`** (default in `src/lib/env.ts`, and the domain that answers). The committed robots.txt/sitemap.xml pointed at `https://gotchicloset.xyz`, which does not respond (connection failure). I treated .xyz as a dead former domain and regenerated both files for .com. If .xyz still has DNS somewhere, a 301 from .xyz to .com should be added at the registrar/Vercel level, but I could not verify or implement that from here.
3. **Vercel serves filesystem matches before the SPA rewrite** (`vercel.json` rewrites `/:path((?!api).*)` to `/index.html`). This is standard Vercel behavior, so `public/llms.txt` will be served as a real text file once deployed. Before this change, `/llms.txt` returned the HTML shell with HTTP 200, which SiteCrawlIQ's llms.txt check reported as a false positive "found".
4. **No commits, pushes, or deploys were made**, per instructions. All changes are working-tree only. The files listed as being edited by the main session (`AuctionGrid.tsx`, `ActivityPage.tsx`, `UserActivityPage.tsx`, `forgeMeta.ts`, `FooterAttribution.tsx`, `env.ts`) were not touched. The single pre-existing `tsc` error (`AuctionGrid.tsx` unused import) belongs to the main session's in-flight work, not to these changes.
5. **Search-volume numbers are not available in this environment** (no keyword tool API). Keyword-map intent/volume ratings are qualitative (high/medium/low/long-tail) based on SERP composition observed via web search, community-size reasoning, and which incumbents rank. The "stale incumbent" flags are based on observed SERPs: most ranking guides are 2021-2023 Polygon-era blog/Medium/wiki content.

## Product and content assumptions

6. **Static shell tradeoff accepted.** Real, crawlable landing content now ships inside `<div id="root">` in `index.html` with inline styles; React replaces it on mount. Users on slow connections may see this content flash briefly before the app hydrates. This was judged acceptable versus the alternative (crawlers seeing an empty page). The FAQ-style H2s in the shell intentionally do NOT get FAQPage JSON-LD, because Google renders JS and would see schema for content not present in the rendered DOM (schema/content mismatch risk). FAQPage schema should ship with the visible FAQ content pages in content-plan.md instead.
7. **Wearable/set counts** stated in copy ("300+ wearables, 150+ sets") come from the repo's own data files (302 wearable pages, 150 set pages in the generated sitemap). If new Forge schematics add items, the copy stays safe because it says "300+"/"150+".
8. **Ecosystem facts** were verified against 2026-visible sources: Base migration completed July 25, 2025 (blog.aavegotchi.com), official dapp at aavegotchi.com, Gotchi lending live on Base, Forge mechanics (smelting returns 90% alloy, half of the loss burned), rarity tier BRS (+1/+2/+5/+10/+20/+50), rarity farming pools split 70/20/10 (rarity/kinship/XP) in recent seasons. Anything I could not verify against a post-migration source is either omitted or marked as "verify" in the content briefs.
9. **The prerender POC (`scripts/prerender.ts`) is deliberately not wired into `pnpm build`.** Rolling it into every Vercel build changes build time and adds Playwright-on-Vercel complexity; the decision to adopt it belongs to the main session. See technical-fixes.md for the tradeoff table.
10. **Audience voice**: landing/answer copy is written plain for AI extraction, with community words (gotchigang, frens, Baazaar spelling) reserved for social/blog contexts, per the prompt.
